# v0.2.9 — Discovery restructured around data (R5/A5), ActionType + C7 rule-bypass fix, container-aware reassessment

A refactor/bug-fix release on top of `0.2.7`, continuing the deep-revision
roadmap from the 2026-07 full-repository review (`docs/reviews/2026-07_full_code_review.md`).
Headline: action discovery is restructured around **data** rather than
hand-repeated instance attributes (R5 + A5), the action-type vocabulary gets a
typed home that fixes a silent expert-rule bypass (C7), and the per-action
reassessment no longer over-subscribes a CPU-limited container.

> **Version note.** `0.2.8` was published to PyPI outside this repository's
> history; this release deliberately skips that number to avoid a version
> collision. `0.2.9` is the next tag cut from this repo after `0.2.7`.

## Highlights

### Discovery restructured around data (deep revisions R5 + A5)

Action discovery no longer carries its per-family outcome as ~40 hand-repeated
instance attributes.

- **`FamilyResult` + `FAMILY_SPECS`** (`action_evaluation/discovery/_results.py`,
  new). One typed `FamilyResult` per family (`identified` / `effective` /
  `ineffective` / `scores` / `params` / `non_convergence`) lives in
  `self.results`, described by a declarative `FAMILY_SPECS` registry that also
  generates the back-compat `@property` bridges for the legacy attribute names
  (`identified_reconnections`, `scores_splits_dict`, `scores_pst_actions`, …) —
  so the family mixins and the existing test suite are untouched.
- **Data-driven assembly + prioritization.** The `__init__` quintuplet
  boilerplate, the PST `getattr` special-casing (PST is now a normal family in
  `self.results`), the 8× `action_scores` literal and the 18 hand-written
  prioritization calls collapse to data-driven loops over
  `ACTION_SCORES_ORDER` / `MIN_PHASE_ORDER` / `FILL_PHASE_ORDER` (one entry per
  family). This makes the previously-duplicated `renewable_curtailment` add
  structurally impossible — it was a **real latent double-add**.
- **Memoised disconnection bounds.** PST scoring's cross-file, order-sensitive
  `_disco_bounds` del/lazy protocol is replaced by a memoised
  `_get_disconnection_bounds()` returning a frozen `DisconnectionBounds`
  (`_pst.py` / `_line_disconnection.py` share it; no call-order dependency).
- **`InjectionDiscoveryBase`** (`_injection_base.py`, new) factors the shared
  overload preamble + influence factor out of the three injection families
  (load shedding / renewable curtailment / redispatch). The divergent cache /
  flow-arity pairing is intentionally kept per-family and documented.
- **`DiscovererProtocol`** (`_protocols.py`, new) declares the shared surface
  the mixins depend on — turning the file split into a real modularization
  (a conformance test asserts the discoverer satisfies it).

Behaviour-preserving: the mock discovery suite is green and `action_scores` is
byte-identical to `0.2.7`.

### ActionType enum + declarative classifier; C7 rule-bypass fix

- **`ActionType` enum + keyword table** (`action_evaluation/action_types.py`,
  new). The 14 action-type tokens get a typed home with `.value` **byte-identical**
  to the historical strings, plus category predicates (`involves_line`,
  `involves_coupling`, `is_open`, `is_topological`, …) that replace scattered
  `"x" in action_type` substring checks. The classifier's description cascade is
  now one ordered `classify_by_description` table.
- **C7 — coupling rule bypass fixed** (`rules.py`). Grid2op-format couplings
  (substation in `content['set_bus']['substations_id']`, no `VoltageLevelId`)
  are now localized backend-agnostically via `_resolve_coupling_subs`.
  Previously they fell through to `"unknown"`, matched none of the five expert
  rules, and escaped filtering entirely — a silent, backend-specific bypass.

### Container-aware reassessment parallelism (2-vCPU fix)

The per-action reassessment (`utils/reassessment.py`) sized its worker pool from
`os.cpu_count()`. Inside a CPU-limited container (e.g. a 2-vCPU cloud box on a
16-core host) that reports the **host** core count, so it spun up to 10 worker
threads — each cloning a full private pypowsybl network — that over-subscribed
the 2 vCPUs and ran *slower* than serial (a reported ~47 s reassessment).

- CPU detection is now container-aware: `_effective_cpu_count()` = min of
  `os.cpu_count()`, scheduler affinity (`os.sched_getaffinity`), and the cgroup
  CPU quota (cgroup v2 `cpu.max` / cgroup v1 `cpu.cfs_quota_us` ÷ `cpu.cfs_period_us`).
- Parallel is engaged only when it pays off. Two new config knobs
  (env-overridable via `EXPERT_OP4GRID_*`):
  - `REASSESSMENT_PARALLEL`: `None` = auto (default), `True` = force parallel,
    `False` = force serial (recommended on 2-vCPU deployments).
  - `REASSESSMENT_MIN_PARALLEL_CORES` (default 4): in auto mode, the minimum
    effective cores required before parallelising.

Behaviour-preserving otherwise; a 4-core `--compare` shows parallel is no faster
than serial there anyway (the per-worker clone tax cancels the concurrency).

### R6 (partial)

All four **C6** `utils/` bugs are verified fixed, and the two previously-untested
data modules (`load_training_data.py` / `load_evaluation_data.py`) got their
first tests. The `utils/` file-move reorganization (`repas/` subpackage,
environment-factory merge, `superposition` / `reassessment` promotion, `__main__`
bodies → `scripts/`) remains to be landed as its own PR.

## Upgrade notes

No breaking API changes. `run_analysis(...)` still returns the same
`AnalysisResult` shape; `action_scores` is byte-identical to `0.2.7`. The legacy
discoverer attribute names (`identified_reconnections`, `scores_splits_dict`,
`scores_pst_actions`, …) continue to work through generated `@property` bridges.
On a CPU-constrained deployment (e.g. a 2-vCPU Space), set
`EXPERT_OP4GRID_REASSESSMENT_PARALLEL=0` to pin reassessment to serial;
otherwise auto-detection now does the right thing.
